# Event Dashboard

A simple desktop dashboard for **GIVE IT ALL** and **Founders Breakfast**. Pulls live data from GoHighLevel (tag counts) and Stripe (actual ticket purchases — name, quantity, amount paid). Can be opened as a local HTML file, or hosted on a subdomain via GitHub Pages so the registration list is reachable from any browser.

## Live hosting on a subdomain (GitHub Pages)

The dashboard can publish itself to a public web address (e.g. `dashboard.yourdomain.com`) so you can open the live registration list from any device — no Mac required to view it.

**One-time setup:**

1. Create a GitHub **Personal Access Token** — go to github.com → Settings → Developer settings → Personal access tokens → *Tokens (classic)* → Generate new token, tick the **`repo`** scope, generate, and copy it.
2. Double-click **`setup-github.command`** in this folder. Enter your GitHub username, paste the token (hidden), accept the default repo name, and type your subdomain (e.g. `dashboard.yourdomain.com`). The script creates the repo, pushes the site, turns on Pages, and prints the DNS record to add.
3. In **Namecheap** → your domain → *Advanced DNS*, add the **CNAME record** the script printed (Host = your subdomain label, Value = `yourusername.github.io`). Save.
4. Wait a few minutes for DNS, then open your subdomain. Done.

**How updates stay live:** `config.json` has `deploy.enabled: true`. Every refresh (manual or scheduled) regenerates the data and automatically commits + pushes it to GitHub, so the live site updates on its own.

**Passcode:** the hosted dashboard asks for a passcode before showing anything. It's set near the top of `index.html` (`const PASSCODE = '...'`) — change that one line to set your own, then refresh to publish.

> **Privacy note:** Free GitHub Pages requires a **public** repository, so the underlying `data.js` and the attendee CSVs are technically downloadable by anyone who finds the URL — the passcode only gates the on-screen dashboard. If you need the data itself protected, ask to switch on encrypted data or move to a private host.

**WhatsApp:** every contact on the dashboard shows a WhatsApp click-to-chat link (built from their phone number). The attendee CSVs include a WhatsApp column too.

## First-time setup

1. **Run a refresh once.** Double-click `refresh.command`. A Terminal window opens, runs the script, closes. This creates `data.js`, the three CSVs, and `stripe-cache.json` (which remembers which Stripe products map to which event so future runs are faster).
2. **Open the dashboard.** Double-click `index.html`. You should see headline numbers, a "Download attendee list" button on each event card, and a bundle row at the bottom. Right-click → Open With → Chrome and pin the tab if you want it permanently open.
3. **Install the schedule.** Double-click `install-schedule.command`. macOS now wakes the refresh at **08:00 every day**, but it only does the actual pull + publish during the **week leading up to the event** (the 4th Thursday of the month) — outside that window it exits immediately. Sleeping Macs skip scheduled jobs.

### Refresh schedule (the lead-up window)

The automatic 8am run is gated to the event's lead-up week so it isn't hammering the APIs all month. The window is controlled in `config.json` under `schedule`:

- `window_enabled` — `true` to limit to the lead-up week, `false` to run every day at 8am.
- `days_before` — how many days before the event the window opens. `6` means the 7 days up to and including the event (opens the prior Friday).
- `include_event_day` — `true` keeps refreshing the morning of the event; `false` stops the day before.

The event date is computed automatically as the **4th Thursday of the current month**, so you don't have to touch the schedule each month. A **manual** `refresh.command` always runs regardless of the window, so you can force an update any time.

## Each month — when the event details change

Open `config.json` and update:

```json
{
  "events": [
    {
      "key": "gia",
      "name": "GIVE IT ALL",
      "date": "2026-06-25",
      "tag": "gia-15-buyer",
      "bundles": [
        { "tag": "gia-15-buyer-4pack", "tickets_per_contact": 4, "label": "4-Pack" }
      ],
      "stripe_product_match": "GIVE IT ALL - Bali"
    },
    {
      "key": "fb",
      "name": "Founders Breakfast",
      "date": "2026-07-07",
      "tag": "fb04",
      "bundles": [],
      "stripe_product_match": "Founders Breakfast"
    }
  ],
  "both_events_tag": "gia15+fb04"
}
```

## Bundle tiers (multi-ticket purchases)

Each event can have one or more **bundles** — GHL tags that mean a buyer purchased N tickets in one transaction. Each bundle has:
- `tag` — the GHL tag applied when this bundle is purchased (e.g. `gia-14-buyer-4pack`)
- `tickets_per_contact` — how many tickets each contact with this tag represents (e.g. 4)
- `label` — display name on the dashboard (e.g. `4-Pack`)

The dashboard headline number is **tickets** (= attendees), not buyers. So a 4-pack buyer counts as 4 in the headline. The CSV gets a "Tickets" column showing the multiplier per row, and "Buyers" is shown in the secondary label so you can see both numbers.

If a contact ends up tagged with both the single tag and a bundle tag (shouldn't happen if your workflow is set up cleanly, but just in case), the script picks the **largest tier** so you don't double-count.

The `stripe_product_match` is a case-insensitive substring of the Stripe product name. As long as the product name in Stripe still contains "GIVE IT ALL" / "Founders Breakfast", you don't need to change it month-to-month — just the dates and the GHL tags.

If you rename the Stripe products, **delete `stripe-cache.json`** so the next refresh re-discovers them.

Save. Run `refresh.command`. Reload the dashboard.

## Guest roster (GIA 4-pack guests)

For events where buyers book extra people in (the GIA "book in your 3 guests" system), the dashboard shows a **Guest Roster** below the event cards: each buyer with their guests nested beneath, plus status ticks **Booked / Profile / Member**.

Turn it on per event in `config.json`:

- `guest_tag` — the GHL tag guests receive (e.g. `gia-15-guest`). Omit it and no roster is built.
- `guest_slots` on a bundle — how many guests that bundle's buyer can book (e.g. `3` on the 4-Pack).
- `status_tags` — which tags mean profile-complete / member (defaults: `profile-complete`, `member`).

How it works:

- Guests are pulled from `guest_tag` and matched to their buyer by the **Booked By** custom field (buyer email), falling back to **Guest Of** (buyer name).
- **Headcount** = buyers + (sum of every buyer's guest slots) = buyers + (4-packs × 3). This is the billing number, shown whether or not guests have booked in yet.
- Empty slots show as "(open guest slot)". Guests whose Booked By doesn't match any buyer appear in a separate "not matched" list.
- The guests are folded into the **main attendee export** (`gia-attendees.csv`) — that file becomes the venue check-in list: one row per head (singles, buyers, and every booked guest), so it stays part of the overall GIA attendee numbers. It's a flat list sorted **alphabetically by name** so the venue can find a person fast; the Type and Guest Of columns preserve who belongs to whom (a guest row shows "Guest" + the buyer who booked them). Columns: Name, Email, Phone, Type (Solo / Buyer (4-Pack) / Guest), Guest Of, Tickets, Profile, Member, GHL Tags — no payment columns (it's a check-in sheet, not a finance report). Open, not-yet-booked guest slots are omitted from the CSV (no name to check in); the **on-screen roster** is where the host sees the full expected party, grouped by buyer with open slots flagged. The headline number on the dashboard card is unchanged — still total attendees/tickets.

Reading the **Booked By / Guest Of / Attendee Type** values requires the script to resolve custom-field IDs; it fetches the location's custom-field definitions automatically each run. If that lookup ever fails, the roster still builds (those columns just come back blank and matching falls back to name).

## How the numbers are calculated

**Headline number** on each event card = **tickets sold via Stripe** (sum of quantities across all completed checkouts in the lookback window).

**Pills below the headline:**

- **Bookings (Stripe)** — number of unique Stripe checkout sessions for that event.
- **Contacts (GHL)** — number of contacts with the GHL tag for that event (single tag + bundle tag combined).
- **Of which: bundle** — Stripe checkouts that included both events (or, if Stripe is unavailable, GHL contacts with the combo tag).

**Bundle row at the bottom** — total bundle tickets and a "Download bundle list" CSV.

If Stripe is unreachable or hasn't been configured, the dashboard falls back to showing GHL contact counts as the headline.

## CSV exports

Every refresh writes three files:

```
gia-attendees.csv   (Date | Name | Email | Tickets | Type | Amount | Currency | Stripe Session ID)
fb-attendees.csv    (same columns)
both-events.csv     (same columns — only the bundle buyers)
```

`Type` = `Solo` (bought only that event) or `Bundle` (bought both at the discounted bundle prices). Bundle buyers appear in both `gia-attendees.csv` and `fb-attendees.csv` (because they're attending both), and also in `both-events.csv`.

Click "Download attendee list" on the dashboard to grab a copy. Or open the file directly from this folder — it'll open in Excel / Numbers.

## Stripe lookback window

`config.json` → `stripe.lookback_days` controls how far back Stripe is queried (default 120 days). Increase if event registration opened earlier than that.

## Manual refresh

Double-click `refresh.command` any time. Reload the dashboard tab afterward (`Cmd+Shift+R` to be safe — browsers cache `data.js` aggressively).

## Troubleshooting

**Dashboard says "No data yet"** — Run `refresh.command`. If it errors, open `refresh.log`.

**Stripe products aren't matching** — Open `refresh.log` and look for `WARNING: no Stripe product matched`. Either rename the Stripe product to include the matching substring, or update `stripe_product_match` in `config.json` to match the current Stripe product name. Then delete `stripe-cache.json` and re-run.

**Counts look wrong** — Open `refresh.log`. It logs each tag count and the Stripe scan stats (e.g. "scanned 47 sessions, 12 matched event products"). Cross-check against Stripe's dashboard.

**Schedule doesn't fire** — `launchctl list | grep ghl-dashboard` in Terminal. If empty, re-run `install-schedule.command`.

**Mac not in Bali timezone** — The schedule fires at the Mac's local 08:00. Either set the Mac's timezone to WITA, or edit the `Hour` value in `com.parallaxx.ghl-dashboard.plist` and re-run the installer.

**Scheduled run did nothing / "outside the lead-up window"** — That's expected outside the event week. Check `refresh.log` for a line like `Schedule window: … outside the lead-up window`. To force a refresh now, double-click `refresh.command` (manual runs ignore the window).

**Token expired** — Update `secrets.json` with a fresh GHL or Stripe key.

**Cloudflare 1010 errors from GHL** — The script sends a browser User-Agent to bypass GHL's bot wall. If GHL ever changes that, we'd need a different UA — let me know.
